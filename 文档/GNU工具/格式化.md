# 格式化 (fmt)

> **对联系统理念**: 以日常汉语为导向，精准表意，规范统一。

## 📋 核心定义

**格式化** 在汉语编程中对应英文命令：**fmt**。

重新格式化文本段落，使其行宽符合指定的要求，同时保持段落的完整性。

**通俗理解**：
- **智能换行**：就像是你写了一篇文章，但每一行的长短不一，看起来很乱。`格式化` 命令就像一个专业的排版编辑，它会根据你设定的宽度，智能地在单词之间换行，让整篇文章看起来整齐美观。
- **保持语义**：与简单的"切断"不同，`格式化` 会尽量保证不在单词中间断开，也不会把原本属于同一段落的内容拆散。

## 💻 用法举例

### 基础用法

| 语境 | 命令示例 |
| :--- | :--- |
| **中文** | `格式化 --宽度 72 文章.txt` |
| **英文** | `fmt -w 72 article.txt` |

### 参数详解

| 参数 | 示例 | 英文参数 | 作用说明 |
| :--- | :--- | :--- | :--- |
| **指定宽度** | `格式化 --宽度 80 文本.txt` | `-w [数字]` / `--width=[数字]` | 设置目标行宽，默认为 75 个字符。 |
| **前缀字符串** | `格式化 --前缀 '> ' 邮件.txt` | `-p [字符串]` / `--prefix=[字符串]` | 只处理以指定字符串开头的行（如邮件引用）。 |
| **统一间距** | `格式化 --统一间距 文本.txt` | `-u` / `--uniform-spacing` | 将单词间的多个空格合并为一个，句尾保留两个空格。 |
| **快速模式** | `格式化 --快速 文本.txt` | `--split-only` | 只拆分过长的行，不合并短行，速度更快。 |
| **帮助信息** | `格式化 帮助信息` | `--help` | 显示完整的帮助文档和所有可用参数。 |
| **版本信息** | `格式化 版本信息` | `--version` | 显示命令的版本号并退出。 |

## ✅ 执行结果介绍

输出内容会被重新排列，每一行的长度都尽量接近你设定的宽度。原本参差不齐的段落会变得像印刷品一样整齐。

## 💻 详细用法举例

### 1. 基础用法 - 整理长段落
```bash
格式化 --宽度 60 草稿.txt
```
**执行结果**：
```
这是一段被重新格式化过的文字。你会发现，
不管原来的行有多长或多短，现在它们都被
整齐地排列在了 60 个字符的宽度以内。这
让阅读体验变得更加舒适。
```
**说明**：`格式化` 会自动识别段落边界，不会把两段话混在一起。

### 2. 处理邮件引用
```bash
格式化 --前缀 '> ' --宽度 70 回复邮件.txt
```
**执行结果**：只处理那些以 `>` 开头的引用行，并保持引用的格式不变。这在整理长篇邮件往来时非常有用。

### 3. 统一空格间距
```bash
格式化 --统一间距 --宽度 80 粗糙文本.txt
```
**执行结果**：如果原文中有些地方用了两个空格，有些地方用了三个，这个命令会把它们全部规范化为一个空格，使排版更专业。

### 4. 保护代码块不被破坏
```bash
格式化 --宽度 1000 混合文档.txt
```
**执行结果**：通过设置一个极大的宽度，你可以让 `格式化` 几乎不进行任何换行操作。这在处理包含代码片段或特殊格式的文档时，可以防止它"好心办坏事"把代码行给折断了。

## 🔍 底层逻辑详解

**格式化** (fmt) 的核心逻辑是 **"单词流重组"** 与 **"贪婪填充"**。

### 1. 它是如何工作的？
- **读取单词**：`格式化` 首先会把输入文本拆解成一个一个的"单词"（以空格或换行符为界）。
- **填充行**：它从第一个单词开始，不断地往当前行里塞单词，直到再塞下一个单词就会超过设定的宽度为止。
- **换行重置**：一旦当前行满了，它就换到下一行，继续重复上述过程。

### 2. 什么是"段落"？
*   **空行分隔**：在 `格式化` 眼里，两个非空行之间的空行就是段落的分界线。
*   **缩进变化**：如果某一行的缩进（开头的空格数）与前一行不同，它通常也被视为新段落的开始。
*   **保护机制**：`格式化` 绝不会跨越段落边界去合并行，这保证了文章的结构不会被破坏。

### 3. 与 fold 的区别
*   **fold (折叠)**：是"暴力"的。它不管你是不是在一个单词的中间，只要到了宽度就一刀切下去。
*   **fmt (格式化)**：是"温柔"的。它会寻找合适的断点（通常是空格），确保切出来的每一行都是完整的单词组合。

### 4. 性能与局限
`格式化` 是一个非常轻量级的工具。它的局限性在于它主要面向纯文本。如果你给它一个包含复杂表格、代码或 Markdown 标记的文件，它可能会把这些结构搞乱。因此，它最适合处理普通的散文、邮件正文或注释文本。

## 📜 Linux 中的定义与发展史

### 1. 打字机时代的排版需求
在计算机普及之前，人们使用打字机撰写文档。由于打字机的纸张宽度固定（通常是 80 列或 72 列），作家和秘书们需要一种方法来确保文字不会超出纸张边缘。`fmt` 的前身就是为此而生的辅助工具。

### 2. Unix V4 的引入
`fmt` 命令最早出现在 1973 年的 Unix V4 中。在那个终端屏幕非常狭小的年代，能够自动调整文本宽度以适应屏幕显示是一项非常实用的功能。

### 3. POSIX 标准的固化
POSIX 标准规定了 `fmt` 的基本行为，包括如何处理前缀、如何识别段落等。这使得它成为了 Unix-like 系统中处理纯文本排版的标配工具。

### 4. 现代系统中的角色
在现代开发环境中，虽然我们有复杂的富文本编辑器，但 `格式化` 依然在以下场景发光发热：
*   **Git 提交信息**：许多团队要求 Git Commit Message 的第一行不超过 50 字符，正文不超过 72 字符，`格式化` 是自动完成这一任务的利器。
*   **代码注释整理**：在重构老旧代码时，用它来批量整理那些写得乱七八糟的多行注释。
*   **纯文本文档发布**：在发布 README.txt 或 CHANGELOG.txt 时，确保它们在任何终端下都能保持良好的可读性。
