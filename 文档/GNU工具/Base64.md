# Base64 (base64)

> **对联系统理念**: 以日常汉语为导向，精准表意，规范统一。

## 📋 核心定义

**Base64** 在汉语编程中对应英文命令：**base64**。

将二进制数据转换为只包含 64 个可打印字符（A-Z, a-z, 0-9, +, /）的文本格式，是互联网上最通用的编码方案。

**通俗理解**：
- **为什么要编码**：就像是你想把一张照片通过只能发文字的短信发出去。Base64 就是把照片的每一个像素点都翻译成字母和数字，对方收到后再按规则翻译回照片。
- **Base64 的特点**：它是互联网世界的“普通话”。虽然它会让文件变大一点（约 33%），但它能确保你的数据在任何古老的邮件系统或网页表单里都不会乱码。

## 💻 用法举例

### 基础用法

| 语境 | 命令示例 |
| :--- | :--- |
| **中文** | `Base64 编码 图片.png` |
| **英文** | `base64 image.png` |

### 参数详解

| 参数 | 示例 | 英文参数 | 作用说明 |
| :--- | :--- | :--- | :--- |
| **解码模式** | `Base64 解码 编码后的文本.txt` | `-d` / `--decode` | 将 Base64 编码的文本还原为原始二进制数据。 |
| **忽略垃圾字符** | `Base64 忽略垃圾 解码 脏数据.txt` | `-i` / `--ignore-garbage` | 解码时跳过非 Base64 字符，提高容错率。 |
| **折行宽度** | `Base64 折行 72 编码 大文件.bin` | `-w [列数]` / `--wrap=[列数]` | 设置每行输出的字符数，0 表示不换行。 |
| **帮助信息** | `Base64 帮助信息` | `--help` | 显示完整的帮助文档和所有可用参数。 |
| **版本信息** | `Base64 版本信息` | `--version` | 显示命令的版本号并退出。 |

## ✅ 执行结果介绍

编码后，原本的二进制数据会变成一串由大小写字母、数字以及 `+` 和 `/` 组成的字符串。解码后，这些数据会完美还原成最初的文件内容。

## 💻 详细用法举例

### 1. 基础用法 - 编码文件
```bash
Base64 编码 签名.jpg
```
**执行结果**：
- 屏幕上会输出一长串类似 `iVBORw0KGgoAAAANSUhEUg...` 的字符。
- 这种格式可以直接嵌入到 HTML 或 CSS 代码中，减少网页的请求次数。

### 2. 解码还原文件
```bash
Base64 解码 编码后的文本.txt > 恢复的图片.jpg
```
**执行结果**：
- 系统将那串字符重新变回原来的图片文件。
- **注意**：如果编码文本里有换行，建议加上 `忽略垃圾` 参数以确保万无一失。

### 3. 快速编码字符串
```bash
echo "你好世界" | Base64 编码
```
**执行结果**：输出 `5L2g5aW95LiW55WM`。这在处理简单的配置项或 API 密钥时非常快捷。

### 4. 生成不换行的紧凑编码
```bash
Base64 折行 0 编码 大数据.bin
```
**执行结果**：所有编码结果挤在一行里，适合在某些不支持换行的配置文件中使用。

## 🔍 底层逻辑详解

**Base64** 的核心逻辑是 **“6比特分组”** 与 **“ASCII 映射”**。

### 1. 为什么是 64？
- **二进制分组**：Base64 每次取 6 个比特（bit）作为一组。因为 2 的 6 次方等于 64，所以正好可以用 64 个不同的字符来表示。
- **字符集选择**：它选用了 26 个大写字母、26 个小写字母、10 个数字，再加上 `+` 和 `/`。这 64 个字符在所有计算机系统中都是通用的 ASCII 字符。

### 2. 编码过程是怎样的？
- **切分**：把原始的 8 比特字节流重新切成每 6 比特一小段。
- **查表**：根据这 6 比特的数值（0-63），去一张固定的表里找对应的字符。
- **填充**：如果最后剩下的比特不足 6 个，就用 `=` 号补齐。这就是为什么你经常在 Base64 字符串末尾看到一两个等号的原因。

### 3. 与 Base32 的对比
*   **Base64**：效率更高，编码后的体积只增加约 33%。它是互联网传输（如电子邮件附件、JSON 数据传输）的事实标准。
*   **Base32**：虽然更抗干扰（不区分大小写、无特殊符号），但体积会增加 60%。所以在空间敏感的场景下，大家更倾向于用 Base64。

### 4. 安全性提醒
*   **不是加密**：Base64 只是“换了一种写法”，任何人都能轻松解码。千万不要用它来存储密码或敏感信息，它只能防止数据在传输过程中因格式问题损坏，不能防止被偷看。

## 📜 Linux 中的定义与发展史

### 1. 电子邮件时代的产物
Base64 最早是为了 MIME（多用途互联网邮件扩展）标准而设计的。在早期的电子邮件系统中，只能传输纯文本，为了发送图片和程序，工程师们发明了 Base64。

### 2. RFC 标准的固化
Base64 的规范定义在 **RFC 4648** 和 **RFC 2045** 中。这些标准确保了无论你是用 Windows、Mac 还是 Linux，编码出来的结果都是一模一样的。

### 3. GNU Coreutils 的集成
GNU 项目组将 `base64` 纳入 Coreutils，使其成为每个 Linux 发行版标配的工具。在现代 DevOps 流程中，它几乎是使用频率最高的编码工具。

### 4. 现代系统中的角色
在现代 Web 开发和云原生环境中，Base64 无处不在：
*   **Data URI**：网页里的图标经常以 `data:image/png;base64,...` 的形式直接写在代码里。
*   **JWT 令牌**：用户登录后的身份凭证（JSON Web Token）通常就是由三段 Base64 编码的字符串组成的。
*   **Kubernetes Secrets**：在 K8s 的配置文件中，所有的密码和密钥都必须先经过 Base64 编码才能存入。
