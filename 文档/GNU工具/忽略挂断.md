# 忽略挂断 (nohup)

> **对联系统理念**: 以日常汉语为导向，精准表意，规范统一。

## 📋 核心定义

**忽略挂断** 在汉语编程中对应英文命令：**nohup**。

运行命令时忽略挂断信号（SIGHUP），使命令在用户注销后继续运行。用于启动长期运行的后台任务。

**通俗理解**：
- **什么是忽略挂断**：当你通过SSH登录服务器运行一个长时间的任务，然后关闭SSH连接时，通常这个任务会被终止。`忽略挂断` 命令可以让任务"免疫"这种终止信号，即使你断开连接，任务也会继续在后台运行。
- **为什么需要它**：有时候你需要运行一些很耗时的任务（如大数据处理、备份、编译等），这些任务可能需要几个小时甚至几天才能完成。你不可能一直保持着SSH连接，所以需要 `忽略挂断` 来确保任务不会因为你的断开而中断。
- **应用场景**：远程服务器任务、长时间运行的进程、后台批处理、自动化脚本、离线任务。

### 输出重定向

默认情况下，`忽略挂断` 会将输出重定向到文件：
- **标准输出和错误**：重定向到 `nohup.out` 文件
- **如果 nohup.out 不可写**：重定向到 `$HOME/nohup.out`

也可以手动指定输出文件：
```bash
忽略挂断 命令 > 输出.log 2>&1 &
```

## 🔍 底层逻辑详解

**忽略挂断** 的核心逻辑是 **"捕获并忽略 SIGHUP 信号"**。

### 1. 工作原理

当执行 `忽略挂断` 命令时：
1. **信号处理**：设置 SIGHUP 信号的处理器为 SIG_IGN（忽略）。
2. **fork子进程**：创建新的进程。
3. **继承信号处理**：子进程继承父进程的信号处理方式。
4. **exec命令**：执行目标命令。
5. **重定向输出**：将stdout和stderr重定向到文件。
6. **返回**：父进程退出，子进程继续在后台运行。

### 2. 系统调用层面

#### 信号忽略
```c
#include <signal.h>

// 忽略 SIGHUP 信号
signal(SIGHUP, SIG_IGN);
```

#### fork 和 exec
```c
#include <unistd.h>

pid_t pid = fork();

if (pid == 0) {
    // 子进程
    // SIGHUP 已经被设置为忽略
    
    // 重定向输出
    int fd = open("nohup.out", O_WRONLY | O_CREAT | O_APPEND, 0600);
    dup2(fd, STDOUT_FILENO);
    dup2(fd, STDERR_FILENO);
    close(fd);
    
    // 执行命令
    execvp(command[0], command);
} else if (pid > 0) {
    // 父进程
    printf("[1] %d\n", pid);
    printf("忽略挂断: 忽略输入并将标准输出追加到 'nohup.out'\n");
    exit(0);
}
```

### 3. SIGHUP 信号

#### 什么是 SIGHUP？
- **名称**：Signal Hang UP（挂断信号）
- **编号**：1
- **来源**：终端控制器检测到连接断开
- **默认行为**：终止进程

#### 何时发送 SIGHUP？
- 用户注销（logout）
- SSH 连接断开
- 终端窗口关闭
- 控制终端消失

#### nohup 的作用
```
正常情况:
用户注销 → 终端发送 SIGHUP → 进程收到 SIGHUP → 进程终止

使用 nohup:
用户注销 → 终端发送 SIGHUP → 进程忽略 SIGHUP → 进程继续运行
```

### 4. 输出重定向

#### 默认行为
```c
// 尝试在当前目录创建 nohup.out
int fd = open("nohup.out", O_WRONLY | O_CREAT | O_APPEND, 0600);

if (fd == -1) {
    // 如果失败，尝试在 HOME 目录创建
    char *home = getenv("HOME");
    char path[PATH_MAX];
    snprintf(path, sizeof(path), "%s/nohup.out", home);
    fd = open(path, O_WRONLY | O_CREAT | O_APPEND, 0600);
}

// 重定向 stdout 和 stderr
dup2(fd, STDOUT_FILENO);
dup2(fd, STDERR_FILENO);
```

#### 手动重定向

### 6. nohup vs screen/tmux

#### nohup
```bash
# 简单直接
忽略挂断 ./任务.sh &
# 缺点：无法重新连接查看输出
```

#### screen
```bash
# 可以分离和重新连接
screen -S 会话名
./任务.sh
# Ctrl+A, D 分离
# screen -r 会话名 重新连接
```

#### tmux
```bash
# 更现代的终端复用器
tmux new -s 会话名
./任务.sh
# Ctrl+B, D 分离
# tmux attach -t 会话名 重新连接
```

### 7. 常见问题

#### 问题1：找不到 nohup.out
**原因**：输出被重定向到其他位置。
**解决**：
```bash
# 检查当前目录
ls -la nohup.out

# 检查 HOME 目录
ls -la ~/nohup.out

# 或者手动指定输出文件
忽略挂断 命令 > 输出.log 2>&1 &
```

#### 问题2：任务仍然被终止
**原因**：任务可能被其他信号终止（如 SIGTERM）。
**解决**：
```bash
# 检查系统日志
dmesg | grep 进程名

# 或使用更 robust 的方法
setsid ./任务.sh &
```

#### 问题3：无法写入 nohup.out
**原因**：权限问题或磁盘空间不足。
**解决**：
```bash
# 检查权限
ls -la nohup.out

# 检查磁盘空间
df -h .

# 手动指定可写的位置
忽略挂断 命令 > /tmp/输出.log 2>&1 &
```

### 8. 最佳实践

### 10. 历史背景

**名称来源**：
- "no hup" = "no hangup"
- 防止进程因挂断信号而终止

**发展历程**：
- **1970年代**：Unix早期就包含 nohup
- **POSIX**：标准化 nohup 的行为
- **现代系统**：仍然是重要的工具

## 📊 使用频率说明

在现代 Linux 系统中，`忽略挂断` (nohup) 的使用频率**中等**。

*   **原因**：
    *   远程服务器管理的常用工具。
    *   被 screen/tmux 部分取代。
    *   在简单场景下仍然非常有用。

*   **替代方案**：
    *   **screen/tmux**：可重新连接的终端会话
    *   **systemd**：专业的服务管理
    *   **Docker**：容器化运行
    *   **disown**：Shell 内置命令
    *   **at/cron**：定时任务调度

*   **推荐使用场景**：
    *   **快速后台任务**：简单的长期任务
    *   **远程服务器**：SSH 会话中的任务
    *   **批处理**：一次性后台任务
    *   **测试环境**：快速启动测试任务
    *   **遗留脚本**：维护使用 nohup 的旧脚本

## 📜 Linux 中的定义与发展史

### 1. Unix 早期的远程连接
1970年代，Unix 通过调制解调器和串行线路连接，连接不稳定，需要 nohup 来保护进程。

### 2. POSIX 标准化
POSIX 标准对 nohup 的行为进行了规范，确保跨平台兼容性。

### 3. SSH 时代的延续
虽然 SSH 比早期的 telnet/rlogin 更稳定，但 nohup 仍然有用，因为：
- 网络仍可能中断
- 用户可能主动断开
- 任务可能需要运行很长时间

### 4. 终端复用器的兴起
- **screen**：1987年发布
- **tmux**：2007年发布
- 提供了更强大的会话管理功能

### 5. 现代服务管理
- **systemd**：成为大多数 Linux 发行版的标准
- **supervisord**：Python 编写的进程管理器
- **Docker**：容器化部署

### 6. uutils 的 Rust 重写
uutils 项目用 Rust 重写了 nohup，提供内存安全保证和跨平台兼容性。

### 7. 云原生时代
在云环境中，nohup 的使用场景有所变化：
- **容器编排**：Kubernetes 管理进程生命周期
- **无服务器**：不需要手动管理进程
- **但仍有用**：在虚拟机和传统服务器上

### 8. 现代应用生态
如今，`忽略挂断` 被应用于：
- **DevOps**：临时任务和测试
- **数据分析**：长时间运行的数据处理
- **机器学习**：模型训练任务
- **系统管理**：维护和备份任务

### 9. 未来的地位
虽然有更先进的工具，但 nohup 由于其简单性和普及率，将继续存在：
- **简单性**：一行命令即可
- **普及性**：所有 Unix-like 系统都有
- **可靠性**：经过几十年验证

未来，nohup 可能会逐渐被更现代化的工具取代，但在简单场景下，它仍然是最快捷的选择。
