# 超时限制 (timeout)

> **对联系统理念**: 以日常汉语为导向，精准表意，规范统一。

## 📋 核心定义

**超时限制** 在汉语编程中对应英文命令：**timeout**。

运行指定的命令，如果在给定时间内未完成则终止它。用于防止命令无限期运行或挂起。

**通俗理解**：
- **什么是超时限制**：就像一个定时器，`超时限制` 命令会给另一个命令设置一个时间上限。如果那个命令在规定时间内完成了，就正常结束；如果超时了，`超时限制` 就会强制终止它。这样可以防止程序卡住或运行太久。
- **为什么需要它**：有时候命令可能会因为各种原因（网络问题、死锁、无限循环等）一直运行下去，占用系统资源。使用 `超时限制` 可以确保命令不会无限制地运行，提高系统的稳定性和可靠性。
- **应用场景**：网络请求、批量处理、自动化脚本、资源管理、防止死锁、定时任务。

## 💻 用法举例

### 基础用法

| 语境 | 命令示例 |
| :--- | :--- |
| **中文** | `超时限制 10 长时间运行的命令` |
| **拼音** | `csxz` |
| **英文** | `timeout 10 long-running-command` |

### 参数详解

| 参数 | 示例 | 英文参数 | 作用说明 |
|------|------|---------|---------|
| **持续时间** | `超时限制 5m 命令` | `DURATION` | 指定超时时间。可以使用后缀：s(秒)、m(分)、h(时)、d(天)。默认为秒。 |
| **杀死信号** | `超时限制 --杀死信号 KILL 10 命令` | `-k DURATION` / `--kill-after=DURATION` | 超时后等待指定时间，然后发送 KILL 信号强制终止。 |
| **保留状态** | `超时限制 --保留状态 命令` | `--preserve-status` | 即使超时也返回命令的退出码，而不是 timeout 的退出码。 |
| **前台运行** | `超时限制 --前台 10 命令` | `--foreground` | 不创建新的进程组，在前台运行。 |
| ** verbose模式** | `超时限制 --详细 10 命令` | `-v` / `--verbose` | 诊断时输出详细信息。 |
| **帮助信息** | `超时限制 --帮助` | `--help` | 显示完整的帮助文档和所有可用参数。 |
| **版本信息** | `超时限制 --版本` | `--version` | 显示命令的版本号并退出。 |

### 时间单位说明

| 单位 | 后缀 | 秒数 | 示例 |
|------|------|------|------|
| **秒** | `s` 或无后缀 | 1 | `10` 或 `10s` = 10秒 |
| **分钟** | `m` | 60 | `5m` = 300秒 |
| **小时** | `h` | 3600 | `1h` = 3600秒 |
| **天** | `d` | 86400 | `1d` = 86400秒 |

## ✅ 执行结果介绍

### 基本执行 - 命令在时限内完成
```bash
超时限制 10 休眠 5
echo $?
```
**执行结果**：`0`
**说明**：命令在10秒内完成，正常退出。

### 命令超时被终止
```bash
超时限制 3 休眠 10
echo $?
```
**执行结果**：`124`
**说明**：命令超时，被 timeout 终止。退出码124表示超时。

### 使用不同时间单位
```bash
超时限制 1m 长时间任务
```
**执行结果**：给命令1分钟的时间。
**说明**：使用分钟作为时间单位。

### 失败情况
如果命令不存在：
```bash
超时限制 5 不存在的命令
```
**执行结果**：
```
超时限制: 无法运行 '不存在的命令': 没有那个文件或目录
```
**退出码**：127

⚠️ **注意**：
- 退出码 124 表示命令超时。
- 退出码 125 表示 timeout 本身失败。
- 退出码 126 表示命令找到但无法执行。
- 退出码 127 表示命令未找到。
- 其他退出码是子命令的退出码（如果使用 --preserve-status）。


- 如果30秒内下载完成：正常结束
- 如果超过30秒：被终止

**说明**：防止下载任务耗时过长。

## 🔍 底层逻辑详解

**超时限制** 的核心逻辑是 **" fork-exec + 定时器 + 信号处理"**。

### 1. 工作原理

当执行 `超时限制` 命令时：
1. **解析参数**：提取超时时间和要执行的命令。
2. **fork子进程**：创建一个新的进程来运行目标命令。
3. **设置定时器**：启动一个计时器。
4. **监控执行**：等待子进程完成或超时。
5. **超时处理**：如果超时，发送信号终止子进程。
6. **返回结果**：返回适当的退出码。

### 2. 系统调用层面

#### 主要流程
```c
#include <sys/types.h>
#include <sys/wait.h>
#include <signal.h>
#include <unistd.h>

pid_t child_pid = fork();

if (child_pid == 0) {
    // 子进程：执行目标命令
    execvp(command[0], command);
} else if (child_pid > 0) {
    // 父进程：设置定时器
    alarm(timeout_seconds);
    
    // 等待子进程
    int status;
    waitpid(child_pid, &status, 0);
    
    // 取消定时器
    alarm(0);
    
    // 返回退出码
    return WEXITSTATUS(status);
}
```

#### 信号处理
```c
void 超时处理(int sig) {
    // 发送 TERM 信号给子进程
    kill(child_pid, SIGTERM);
    
    // 如果指定了 --kill-after
    if (kill_after > 0) {
        alarm(kill_after);
        // 下次超时时发送 KILL
    } else {
        exit(124);
    }
}

signal(SIGALRM, 超时处理);
```

### 3. 时间单位的解析

```c
long 解析时间(const char *str) {
    char *endptr;
    long value = strtol(str, &endptr, 10);
    
    switch (*endptr) {
        case 's': return value;
        case 'm': return value * 60;
        case 'h': return value * 3600;
        case 'd': return value * 86400;
        default:  return value;  // 默认秒
    }
}
```

### 4. 退出码的含义

| 退出码 | 含义 |
|--------|------|
| **0-123** | 子命令的退出码 |
| **124** | 命令超时 |
| **125** | timeout 本身失败 |
| **126** | 命令找到但无法执行 |
| **127** | 命令未找到 |
| **128+** | 命令被信号终止（128+信号编号） |

### 5. 与相关命令的对比

| 命令 | 功能 | 优势 | 劣势 |
|------|------|------|------|
| **timeout** | 超时控制 | 简单、标准 | 仅时间控制 |
| **timelimit** | 类似功能 | 更多选项 | 非标准 |
| **ulimit -t** | CPU时间限制 | 内核级别 | 仅CPU时间 |
| **watchdog** | 看门狗 | 持续监控 | 复杂 |
| **systemd** | 服务管理 | 集成度高 | 仅systemd服务 |

### 6. 信号处理流程

```
超时发生
   ↓
发送 SIGTERM
   ↓
等待 --kill-after 秒
   ↓
如果还在运行
   ↓
发送 SIGKILL
   ↓
强制终止
```

### 7. 常见问题

#### 问题1：子进程忽略 SIGTERM
**原因**：某些程序捕获但不处理 SIGTERM。
**解决**：
```bash
# 使用 --kill-after 强制终止
超时限制 --杀死信号 5 30 命令
```

#### 问题2：子进程产生孤儿进程
**原因**：子进程fork后父进程退出。
**解决**：
```bash
# 使用进程组
超时限制 --前台 30 命令
```

#### 问题3：时间精度问题
**原因**：alarm() 只支持秒级精度。
**解决**：对于高精度需求，使用其他工具。

## 📜 Linux 中的定义与发展史

### 1. 早期 Unix 的需求
早期的Unix系统缺乏标准的超时控制工具，脚本作者需要自己实现超时逻辑。

### 2. GNU Coreutils 的引入
GNU项目在Coreutils包中引入了 timeout 命令，填补了这一空白。

### 3. POSIX 标准化
虽然 timeout 不是 POSIX 标准命令，但它已成为事实标准，被广泛采用。

### 4. systemd 的影响
systemd 引入了自己的超时机制（TimeoutStartSec等），但 timeout 命令仍然有用。

### 5. 云原生时代
在容器和微服务架构中，超时控制变得更加重要：
- **健康检查**：快速检测服务状态
- **服务网格**：请求超时控制
- **编排系统**：任务超时管理

### 6. uutils 的 Rust 重写
uutils项目用Rust重写了 timeout，提供内存安全保证和更好的性能。

### 7. 现代应用生态
如今，`超时限制` 被应用于：
- **DevOps**：CI/CD 流水线
- **微服务**：服务间调用超时
- **数据库**：查询超时控制
- **网络编程**：连接和请求超时
- **自动化测试**：测试用例超时

### 8. 未来的地位
随着分布式系统和微服务架构的普及，超时控制变得越来越重要。timeout 命令将继续发挥重要作用：
- **必要性**：防止资源泄漏的基本手段
- **通用性**：适用于任何命令
- **标准化**：GNU Coreutils 的一部分

未来，timeout 可能会与更先进的可观测性和弹性模式集成，但其核心的超时控制功能将继续存在。
