# Base32 (base32)

> **对联系统理念**: 以日常汉语为导向，精准表意，规范统一。

## 📋 核心定义

**Base32** 在汉语编程中对应英文命令：**base32**。

将二进制数据转换为只包含 32 个可打印字符（A-Z 和 2-7）的文本格式，便于在不支持二进制的系统中传输。

**通俗理解**：
- **为什么要编码**：就像是你想把一幅画通过电报发出去，但电报只能发文字。于是你把画的颜色、形状都转化成字母和数字，对方收到后再按规则还原成画。
- **Base32 的特点**：它只用大写字母 A-Z 和数字 2-7，避开了容易混淆的字符（如 0 和 O，1 和 l），非常适合人工抄写或语音传达。

## 💻 用法举例

### 基础用法

| 语境 | 命令示例 |
| :--- | :--- |
| **中文** | `Base32 编码 秘密文件.bin` |
| **英文** | `base32 secret_file.bin` |

### 参数详解

| 参数 | 示例 | 英文参数 | 作用说明 |
| :--- | :--- | :--- | :--- |
| **解码模式** | `Base32 解码 编码后的文本.txt` | `-d` / `--decode` | 将 Base32 编码的文本还原为原始二进制数据。 |
| **忽略垃圾字符** | `Base32 忽略垃圾 解码 脏数据.txt` | `-i` / `--ignore-garbage` | 解码时跳过非 Base32 字符，提高容错率。 |
| **不填充等号** | `Base32 不填充 编码 数据.bin` | `--no-pad` | 编码结果末尾不加 `=` 填充符，使输出更紧凑。 |
| **帮助信息** | `Base32 帮助信息` | `--help` | 显示完整的帮助文档和所有可用参数。 |
| **版本信息** | `Base32 版本信息` | `--version` | 显示命令的版本号并退出。 |

## ✅ 执行结果介绍

编码后，原本乱码般的二进制数据会变成一串由大写字母和数字组成的“天书”。解码后，这些数据会完美还原成最初的文件内容。

## 💻 详细用法举例

### 1. 基础用法 - 编码文件
```bash
Base32 编码 密码本.bin
```
**执行结果**：
- 屏幕上会输出一长串类似 `JBSWY3DPEHPK3PXP...` 的字符。
- 这些字符可以安全地通过电子邮件、短信甚至口头传达。

### 2. 解码还原文件
```bash
Base32 解码 编码后的文本.txt > 恢复的文件.bin
```
**执行结果**：
- 系统将那串“天书”重新变回原来的二进制文件。
- **注意**：如果编码文本里有换行或空格，建议加上 `忽略垃圾` 参数。

### 3. 管道操作（实时编码）
```bash
查看 密钥.key | Base32 编码
```
**执行结果**：直接读取文件内容并转换成 Base32 格式，无需生成中间文件。

### 4. 处理带杂质的编码文本
```bash
Base32 忽略垃圾 解码 "JBSW Y3DP EHPK 3PXP"
```
**执行结果**：即使字符串里有空格或换行，它也能智能识别并正确解码。

## 🔍 底层逻辑详解

**Base32** 的核心逻辑是 **“进制转换”** 与 **“字符映射”**。

### 1. 为什么是 32？
- **二进制分组**：Base32 每次取 5 个比特（bit）作为一组。因为 2 的 5 次方等于 32，所以正好可以用 32 个不同的字符来表示。
- **字符集选择**：它选用了 26 个大写字母（A-Z）和 6 个数字（2-7）。为什么不选 0, 1, 8, 9？因为在很多字体里，0 和 O、1 和 l、8 和 B 长得太像了，容易看走眼。

### 2. 编码过程是怎样的？
- **切分**：把原始的二进制数据切成每 5 个比特一小段。
- **查表**：根据这 5 个比特的数值（0-31），去一张固定的表里找对应的字母或数字。
- **填充**：如果最后剩下的比特不足 5 个，就用 `=` 号补齐，确保接收方能对齐数据。

### 3. 与 Base64 的对比
*   **Base64**：用 64 个字符，效率更高（膨胀率约 33%），但包含了小写字母和特殊符号（+, /），在某些严格的环境（如文件名、URL）下会出问题。
*   **Base32**：虽然效率稍低（膨胀率约 60%），但它**不区分大小写**且**没有特殊符号**，在人工输入或跨平台传输时更可靠。

### 4. 应用场景
*   **TOTP 双因素认证**：你在绑定 Google Authenticator 时看到的那串长长的密钥，通常就是 Base32 编码的。
*   **DNS 记录**：由于 DNS 对字符限制很严，Base32 常被用来在域名里嵌入二进制数据。

## 📜 Linux 中的定义与发展史

### 1. 早期互联网的需求
在互联网早期，很多邮件系统和网络协议只支持 7 位的 ASCII 码。为了传输图片、程序等二进制文件，人们发明了各种编码方案，Base32 和 Base64 就是其中的佼佼者。

### 2. RFC 标准的制定
Base32 的正式规范定义在 **RFC 4648** 中。这份标准不仅规定了字符集，还详细描述了填充规则和换行处理方式，确保了不同软件之间的兼容性。

### 3. GNU Coreutils 的集成
GNU 项目组将 `base32` 纳入 Coreutils，使其成为 Linux 系统自带的工具箱之一。在此之前，用户通常需要安装额外的加密工具包才能进行此类操作。

### 4. 现代系统中的角色
在现代云原生和微服务架构中，Base32 依然活跃：
*   **配置注入**：在 Kubernetes 中，敏感信息（Secrets）常以 Base32 或 Base64 形式存储在 YAML 文件里。
*   **短链接生成**：一些 URL 缩短服务利用 Base32 的特性，生成既短又不容易出错的识别码。
