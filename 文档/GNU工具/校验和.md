# 校验和 (sum)

> **对联系统理念**: 以日常汉语为导向，精准表意，规范统一。

## 📋 核心定义

**校验和** 在汉语编程中对应英文命令：**sum**。

计算文件的BSD风格校验和和块数。这是一个古老的校验和算法，主要用于向后兼容，现代应用应使用md5sum或sha256sum等更安全的算法。

**通俗理解**：
- **什么是校验和**：就像给文件生成一个"老旧的数字指纹"，`校验和` 命令会使用一个简单的算法计算文件的校验和，并显示文件占用的块数。这个命令非常古老，算法简单且不安全，现在主要用于兼容旧系统和旧脚本。
- **为什么需要它**：虽然sum已经过时，但你可能在维护旧系统、处理遗留脚本或与老式Unix系统交互时遇到它。了解sum有助于理解校验和技术的发展历程，以及在需要兼容性时的正确用法。
- **应用场景**：遗留系统维护、旧脚本兼容、教育演示、与老式Unix系统交互、历史研究。

## 💻 用法举例

### 基础用法

| 语境 | 命令示例 |
| :--- | :--- |
| **中文** | `校验和 文件.txt` |
| **拼音** | `xyh` |
| **英文** | `sum 文件.txt` |

### 参数详解

| 参数 | 示例 | 英文参数 | 作用说明 |
|------|------|---------|---------|
| **System V算法** | `校验和 --sysv 文件` | `-s` / `--sysv` | 使用System V校验和算法（16位）。 |
| **BSD算法** | `校验和 --bsd 文件` | `-r` / `--bsd` | 使用BSD校验和算法（默认，16位）。 |
| **帮助信息** | `校验和 --帮助` | `--help` | 显示完整的帮助文档。 |
| **版本信息** | `校验和 --版本` | `--version` | 显示版本号并退出。 |

### 算法对比

| 算法 | 位数 | 输出格式 | 说明 |
|------|------|---------|------|
| **BSD** | 16位 | 校验和 块数 文件名 | 默认算法 |
| **System V** | 16位 | 校验和 块数 文件名 | 替代算法 |

### 输出格式

```
校验和 块数 文件名
```

- **校验和**：16位无符号整数（0-65535）
- **块数**：文件占用的512字节块数（向上取整）
- **文件名**：处理的文件名

⚠️ **重要警告**：
- **sum 已过时**：不应用于任何安全或完整性验证场景。

## 🔍 底层逻辑详解

**校验和** 的核心逻辑是 **"实现简单的16位校验和算法"**。

### 1. BSD 校验和算法

#### 算法步骤
```c
unsigned short bsd_checksum(FILE *fp) {
    unsigned short checksum = 0;
    int c;
    
    while ((c = fgetc(fp)) != EOF) {
        // 循环右移1位
        checksum = ((checksum >> 1) | ((checksum & 1) << 15)) & 0xFFFF;
        // 加上当前字节
        checksum = (checksum + c) & 0xFFFF;
    }
    
    return checksum;
}
```

### 2. System V 校验和算法

#### 算法步骤
```c
unsigned short sysv_checksum(FILE *fp) {
    unsigned short checksum = 0;
    int c;
    
    while ((c = fgetc(fp)) != EOF) {
        checksum = (checksum + c) & 0xFFFF;
    }
    
    return checksum;
}
```

### 3. 块数计算

```c
long 计算块数(long 文件大小) {
    const int 块大小 = 512;  // BSD传统块大小
    return (文件大小 + 块大小 - 1) / 块大小;  // 向上取整
}
```

### 4. 算法弱点

#### 碰撞容易
- **16位空间**：只有65536种可能
- **生日攻击**：约256个文件就有50%碰撞概率
- **无雪崩效应**：输入微小变化不一定导致输出巨大变化

#### 安全性差
- **可预测**：算法简单，容易逆向
- **无抗碰撞性**：可以故意构造碰撞
- **无抗原像性**：可以找到产生特定校验和的输入

### 5. 与现代算法的对比

| 特性 | sum | md5sum | sha256sum |
|------|-----|--------|-----------|
| **位数** | 16 | 128 | 256 |
| **碰撞阻力** | 极弱 | 弱 | 强 |
| **速度** | 快 | 中 | 较慢 |
| **安全性** | ❌ | ⚠️ | ✅ |
| **推荐度** | 不推荐 | 谨慎 | 推荐 |

### 6. 系统调用层面

```c
#include <stdio.h>
#include <unistd.h>

int main(int argc, char *argv[]) {
    FILE *fp;
    unsigned short checksum;
    long 块数;
    struct stat st;
    
    for (int i = 1; i < argc; i++) {
        fp = fopen(argv[i], "r");
        if (!fp) {
            perror(argv[i]);
            continue;
        }
        
        fstat(fileno(fp), &st);
        块数 = (st.st_size + 511) / 512;
        
        checksum = bsd_checksum(fp);  // 或 sysv_checksum
        
        printf("%u %ld %s\n", checksum, 块数, argv[i]);
        fclose(fp);
    }
    
    return 0;
}
```

## 📊 使用频率说明

在现代 Linux 系统中，`校验和` (sum) 的使用频率**极低**。

*   **原因**：
    *   sum已经过时且不安全。
    *   现代工具（md5sum、sha256sum）完全取代了它。
    *   仅用于遗留系统兼容性。

*   **替代方案**：
    *   **md5sum**：基本的完整性检查
    *   **sha256sum**：推荐的标准
    *   **sha512sum**：高安全性需求
    *   **b2sum**：更快的BLAKE2
    *   **cksum**：POSIX标准的CRC校验

*   **推荐使用场景**：
    *   **遗留系统**：维护旧系统
    *   **教育目的**：讲解校验和演化
    *   **历史研究**：技术研究
    *   **兼容性测试**：跨平台兼容
    *   **迁移过渡**：从sum迁移到现代算法的过程

## 📜 Linux 中的定义与发展史

### 1. 早期Unix
1970年代，Unix引入了简单的checksum概念。

### 2. BSD和System V的分歧
1980年代，BSD和System V实现了不同的校验和算法。

### 3. POSIX标准化
POSIX定义了cksum命令（使用CRC32），sum逐渐成为遗留工具。

### 4. 现代哈希算法的出现
- **1990年代**：MD5、SHA-1
- **2000年代**：SHA-2家族
- **2010年代**：SHA-3、BLAKE2

### 5. GNU Coreutils 的保留
GNU项目保留了sum命令，主要用于向后兼容。

### 6. uutils 的 Rust 重写
uutils项目用Rust重写了sum。

### 7. 现代应用生态
如今，`校验和` (sum) 几乎不再使用：
- **遗留系统**：极少数旧系统
- **教育**：计算机科学教学
- **历史**：技术研究

### 8. 淘汰趋势
- **2000年后**：大多数新系统不再使用sum
- **2010年后**：文档开始明确标注为过时
- **未来**：可能被移除或标记为废弃

### 9. 未来的地位
sum将继续存在，但仅作为历史遗迹：
- **兼容性**：保持与旧脚本的兼容
- **教育**：讲解技术发展史
- **研究**：密码学演进研究

未来，sum可能会在某个版本的coreutils中被标记为deprecated，最终可能被移除。现在就应该避免在任何新代码中使用它。
