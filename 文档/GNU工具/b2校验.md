# b2校验 (b2sum)

> **对联系统理念**: 以日常汉语为导向，精准表意，规范统一。

## 📋 核心定义

**b2校验** 在汉语编程中对应英文命令：**b2sum**。

计算和验证文件的 BLAKE2 哈希校验和，确保数据完整性与真实性。

**通俗理解**：
- **数字指纹生成器**：就像每个人的指纹都是独一无二的一样，`b2校验` 能为任何文件生成一个独一无二的"数字指纹"（哈希值）。只要文件内容有一丁点变化，这个指纹就会完全不同。
- **安全升级版**：它是传统 MD5 和 SHA 系列的现代化替代品，速度更快、安全性更高，特别适合大文件校验和密码学应用。
- **可定制长度**：与其他校验工具不同，你可以自定义输出哈希值的长度，灵活适应不同场景的需求。

## 💻 用法举例

### 基础用法

| 语境 | 命令示例 |
| :--- | :--- |
| **中文** | `b2校验 文件.txt` |
| **拼音** | `b2jy` |
| **英文** | `b2sum file.txt` |

### 参数详解

| 参数 | 示例 | 英文参数 | 作用说明 |
| :--- | :--- | :--- | :--- |
| **校验模式** | `b2校验 --校验 校验和文件.b2` | `-c` / `--check` | 从文件中读取 BLAKE2 校验和并进行验证，确认文件是否被篡改。 |
| **哈希长度** | `b2校验 --哈希长度 32 数据.bin` | `-l [长度]` / `--length=[N]` | 指定输出的哈希值位数（范围1-64），默认为64位（512比特）。较短的长度适合快速校验，较长的长度提供更高安全性。 |
| **文本模式** | `b2校验 --文本模式 文档.txt` | `-t` / `--text` | 以文本模式读取文件（默认模式），适用于普通文本文件。 |
| **二进制模式** | `b2校验 --二进制模式 程序.exe` | `-b` / `--binary` | 以二进制模式读取文件，在 Windows 系统中区分文本和二进制换行符。 |
| **标签格式** | `b2校验 --标签格式 镜像.iso` | `--tag` | 使用 BSD 风格的标签格式输出（如 `BLAKE2b (文件名) = 哈希值`），便于与其他工具兼容。 |
| **零字节分隔** | `b2校验 --零字节分隔 文件列表.txt` | `-z` / `--zero` | 使用空字符（NULL）而非换行符分隔输出，适合处理包含特殊字符的文件名。 |
| **静默模式** | `b2校验 --校验 --静默模式 校验和.b2` | `--quiet` | 在校验模式下，只报告失败的文件，不显示成功的信息。 |
| **状态模式** | `b2校验 --校验 --状态模式 校验和.b2` | `--status` | 不输出任何信息，仅通过退出码表示校验结果（0=成功，1=失败）。 |
| **严格模式** | `b2校验 --校验 --严格模式 校验和.b2` | `--strict` | 对校验和文件的格式进行严格检查，遇到任何格式错误都会报错。 |
| **忽略缺失** | `b2校验 --校验 --忽略缺失 校验和.b2` | `--ignore-missing` | 在校验时，如果某些文件不存在则跳过而不报错。 |
| **警告模式** | `b2校验 --校验 --警告模式 校验和.b2` | `-w` / `--warn` | 当校验和文件格式不正确时发出警告（默认行为）。 |
| **原始输出** | `b2校验 --原始输出 数据.dat` | `--raw` | 以原始二进制形式输出哈希值，而非十六进制字符串。 |
| **Base64输出** | `b2校验 --base64输出 数据.dat` | `--base64` | 以 Base64 编码形式输出哈希值，适合在网络传输中使用。 |
| **帮助信息** | `b2校验 --帮助` | `--help` | 显示完整的帮助文档和所有可用参数。 |
| **版本信息** | `b2校验 --版本` | `--version` | 显示命令的版本号并退出。 |

## ✅ 执行结果介绍

### 计算模式
默认情况下，输出格式为：`哈希值  文件名`

例如：
```
a8add4bdddfd93e4877d2746e62817b116364a1fa7bc148d95090bc7333b3673f82401cf7aa2e4cb1ecd90296e3f14cb5413f8ed77be73045b13914cdcd6a918  重要文档.pdf
```

### 校验模式
校验成功后显示：
```
重要文档.pdf: 确定
```

校验失败时显示：
```
重要文档.pdf: 失败
b2校验: 警告: 1 个校验和不匹配
```

⚠️ **注意**：校验和是单向的，无法从哈希值还原原始文件内容。

## 💻 详细用法举例

### 1. 基础用法 - 生成文件校验和
```bash
b2校验 软件安装包.tar.gz
```
**执行结果**：
```
c6f35011e3c0e1e0e3b5f1a3d2c4b5a6e7f8d9c0b1a2e3f4d5c6b7a8e9f0d1c2b3a4e5f6d7c8b9a0e1f2d3c4b5a6e7f8d9c0b1a2e3f4d5c6b7a8e9f0d1  软件安装包.tar.gz
```
**说明**：生成该安装包的 BLAKE2 哈希值，可用于后续验证文件完整性。

### 2. 保存校验和到文件
```bash
b2校验 项目源码.zip > 源码校验和.b2
```
**执行结果**：将哈希值保存到 `源码校验和.b2` 文件中。
**说明**：这是分发文件时的标准做法，让用户可以下载后验证文件是否完整。

### 3. 验证文件完整性
```bash
b2校验 --校验 源码校验和.b2
```
**执行结果**：
```
项目源码.zip: 确定
```
**说明**：从校验和文件中读取哈希值并与当前文件对比，确认文件未被篡改或损坏。

### 4. 自定义哈希长度
```bash
b2校验 --哈希长度 32 配置文件.json
```
**执行结果**：
```
e3b0c44298fc1c149afbf4c8996fb92427ae41e4649b934ca495991b7852b855  配置文件.json
```
**说明**：输出32位（256比特）的哈希值，比默认的64位更短，适合空间受限的场景。

### 5. 批量校验多个文件
```bash
b2校验 文件1.txt 文件2.txt 文件3.txt > 批量校验和.b2
b2校验 --校验 批量校验和.b2
```
**执行结果**：
```
文件1.txt: 确定
文件2.txt: 确定
文件3.txt: 确定
```
**说明**：一次性为多个文件生成校验和，然后批量验证，非常适合备份验证。

### 6. 管道组合使用
```bash
查看 敏感数据.csv | b2校验
```
**执行结果**：
```
a1b2c3d4e5f6...  -
```
**说明**：从标准输入读取数据并计算哈希值，适合流式数据处理。

### 7. 使用标签格式（BSD风格）
```bash
b2校验 --标签格式 数据库备份.sql
```
**执行结果**：
```
BLAKE2b (数据库备份.sql) = a8add4bdddfd93e4877d2746e62817b116364a1fa7bc148d95090bc7333b3673f82401cf7aa2e4cb1ecd90296e3f14cb5413f8ed77be73045b13914cdcd6a918
```
**说明**：这种格式更容易被其他工具解析，特别是在跨平台场景中。

### 8. 静默模式用于脚本
```bash
b2校验 --校验 --状态模式 自动化校验.b2 && 显示 "校验通过" || 显示 "校验失败"
```
**执行结果**：无输出，但会根据校验结果执行不同的命令。
**说明**：在自动化脚本中，通过退出码判断校验结果，无需解析文本输出。

## 🔍 底层逻辑详解

**b2校验** 基于 BLAKE2 哈希算法，这是一种现代化的密码学哈希函数，在设计上兼顾了速度与安全性。

### 1. BLAKE2 算法的核心优势
- **速度极快**：在现代 CPU 上，BLAKE2b 的速度可达 MD5 的 1.5 倍，SHA-256 的 3 倍以上。它充分利用了 CPU 的并行处理能力。
- **安全性高**：没有已知的碰撞攻击（两个不同文件产生相同哈希值）或原像攻击（从哈希值反推文件内容）。
- **可配置性**：支持 1 到 64 字节（8 到 512 比特）的可变输出长度，适应不同安全需求。
- **密钥支持**：可以作为消息认证码（MAC）使用，提供额外的身份验证层。

### 2. 哈希计算的工作原理
- **分块处理**：文件被分割成固定大小的块（BLAKE2b 为 128 字节），逐块进行处理。
- **压缩函数**：每一块数据都与前一块的哈希状态混合，通过复杂的数学运算（包括加法、异或、旋转等）生成新的哈希状态。
- **最终化**：处理完所有块后，对最终的哈希状态进行压缩，生成固定长度的输出。
- **雪崩效应**：输入数据的任何微小变化（哪怕只改了一个比特），都会导致输出哈希值发生巨大变化（约 50% 的比特位会翻转）。

### 3. 为什么选择 BLAKE2 而非 MD5/SHA-1？
- **MD5 已不安全**：早在 2004 年就被证明存在碰撞漏洞，攻击者可以故意制造两个不同文件但具有相同的 MD5 值。
- **SHA-1 已被攻破**：2017 年 Google 展示了实际的 SHA-1 碰撞攻击，不再适合安全场景。
- **SHA-256 较慢**：虽然安全，但在软件实现上比 BLAKE2 慢很多。
- **BLAKE2 的平衡**：在保持与 SHA-3 相当的安全性的同时，提供了接近 MD5 的速度。

### 4. 校验和验证流程
1. **读取校验文件**：解析 `.b2` 文件中的哈希值和文件名。
2. **重新计算**：对指定文件再次运行 BLAKE2 算法。
3. **逐位对比**：将新计算的哈希值与存储的哈希值进行比对。
4. **结果判定**：完全匹配则"确定"，否则"失败"。

### 5. 文本模式 vs 二进制模式
- **文本模式**（默认）：在某些系统上会将换行符 `\r\n` 转换为 `\n`，确保跨平台一致性。
- **二进制模式**：原样读取所有字节，不做任何转换。在 Linux 上两者通常没有区别，但在 Windows 上会影响结果。

## 📊 使用频率说明

在现代 Linux 系统和软件开发中，`b2校验` (b2sum) 的使用频率**中等偏高**，且呈上升趋势。

*   **原因**：
    *   BLAKE2 是较新的算法（2012 年发布），许多旧系统和脚本仍在使用 MD5 或 SHA-256。
    *   随着对安全性和性能要求的提高，越来越多的新项目开始采用 BLAKE2。
    *   在区块链、密码学库和高性能计算领域，BLAKE2 已成为首选。

*   **替代方案**：
    *   **md5校验**：速度快但不安全，仅适合非安全场景的快速校验。
    *   **sha256校验**：安全性好且广泛支持，是目前最常用的校验工具。
    *   **sha512校验**：提供更高的安全级别，但计算速度较慢。

*   **推荐使用场景**：
    *   **高性能需求**：需要频繁计算大文件哈希值时（如视频处理、大数据集）。
    *   **现代安全应用**：开发新的加密协议或身份验证系统。
    *   **资源受限环境**：BLAKE2 的实现更简洁，占用内存更少。
    *   **未来兼容性**：为新项目选择长期安全的哈希算法。

## 📜 Linux 中的定义与发展史

### 1. BLAKE 竞赛的诞生
2007 年，美国国家标准与技术研究院（NIST）发起了一场全球哈希算法竞赛，旨在寻找 SHA-2 的继任者（即 SHA-3）。来自全球的团队提交了 64 个候选算法。

### 2. BLAKE2 的问世
2012 年，在 BLAKE（SHA-3 决赛选手之一）的基础上，Jean-Philippe Aumasson 等人发布了 BLAKE2。它不是参赛 SHA-3 的，而是为了提供一个比现有算法更快、更安全的替代方案。

### 3. GNU Coreutils 的整合
随着 BLAKE2 在开源社区的普及，GNU Coreutils 在 2016 年左右引入了 `b2sum` 命令，使其成为标准校验工具家族的一员。

### 4. 现代应用生态
如今，BLAKE2 已被广泛应用于：
- **密码学库**：libsodium、OpenSSL、BoringSSL 等都内置了 BLAKE2 支持。
- **区块链**：Zcash 等加密货币使用 BLAKE2b 作为其哈希函数的核心组件。
- **文件去重**：Dropbox、rsync 等工具利用 BLAKE2 的高速特性进行文件指纹识别。
- **操作系统**：Linux 内核在某些子系统中采用 BLAKE2 进行数据完整性校验。

### 5. 未来的地位
随着量子计算的发展和对传统哈希算法的持续攻击，BLAKE2 因其简洁的设计和强大的安全性，被认为是未来十年内最可靠的哈希算法之一。对于新开发的系统，选择 `b2校验` 是一个面向未来的明智决策。
