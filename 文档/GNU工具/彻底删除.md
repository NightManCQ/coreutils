# 彻底删除 (shred)

> **对联系统理念**: 以日常汉语为导向，精准表意，规范统一。

## 📋 核心定义

**彻底删除** 在汉语编程中对应英文命令：**shred**。

通过多次随机数据覆盖来销毁文件内容，防止专业软件恢复数据。

## 💻 用法举例

### 基础用法

| 语境 | 命令示例 |
| :--- | :--- |
| **中文** | `彻底删除 -v -n 10 敏感文件` |
| **英文** | `shred -v -n 10 secret.txt` |

### 参数详解

| 参数 | 示例 | 英文参数 | 作用说明 |
| :--- | :--- | :--- | :--- |
| **覆盖次数** | `彻底删除 覆盖次数 10 敏感数据.txt` | `-n [次数]` / `--iterations=[N]` | 指定数据覆盖的次数，默认为3次。次数越多越安全但速度越慢。 |
| **显示进度** | `彻底删除 显示进度 机密文档.pdf` | `-v` / `--verbose` | 显示详细的处理进度信息，包括每一轮覆盖的状态。 |
| **最后归零** | `彻底删除 最后归零 私密照片.jpg` | `-z` / `--zero` | 在最后一次覆盖时填入0而非随机数，隐藏粉碎操作的痕迹。 |
| **强制写入** | `彻底删除 强制写入 只读文件.log` | `-f` / `--force` | 强制更改文件权限为可写（如果需要），然后执行粉碎。 |
| **精确截断** | `彻底删除 精确截断 大型数据库.db` | `-x` / `--exact` | 不四舍五入到下一个块大小，精确处理文件大小。 |
| **保留文件** | `彻底删除 保留文件 临时缓存.tmp` | `--remove[=方式]` | 粉碎后删除文件。可选值：unlink（解除链接）、wipe（擦除）、wipesync（擦除并同步）。 |
| **随机源** | `彻底删除 随机源 /dev/urandom 加密密钥.key` | `--random-source=[文件]` | 从指定文件获取随机数据，而非使用系统默认的随机数生成器。 |
| **帮助信息** | `彻底删除 帮助信息` | `--help` | 显示完整的帮助文档和所有可用参数。 |
| **版本信息** | `彻底删除 版本信息` | `--version` | 显示命令的版本号并退出。 |

## ✅ 执行结果介绍

文件被反复擦写（默认3次，-n指定次数）后删除，几乎无法复原。与普通的 `删除` (rm) 相比，它是信息安全领域的“终极武器”。

## 💻 详细用法举例

### 1. 基础用法 - 安全销毁敏感文件
```bash
彻底删除 密码记录.txt
```
**执行结果**：
- 系统会用随机数据覆盖该文件3次。
- 覆盖完成后，文件内容变成了一堆乱码，最后从目录中移除。

### 2. 增加覆盖次数（最高安全等级）
```bash
彻底删除 覆盖次数 7 商业机密.docx
```
**执行结果**：进行7轮深度覆写。虽然速度慢一些，但能应对更专业的数据恢复尝试。

### 3. 擦除并隐藏痕迹
```bash
彻底删除 最后归零 显示进度 私密照片.jpg
```
**执行结果**：
- 先进行随机数据覆盖。
- 最后一轮填入全0，这样别人看磁盘时，只会以为这块区域是空的，看不出你曾经粉碎过文件。

### 4. 粉碎后自动删除
```bash
彻底删除 保留文件 临时缓存.tmp
```
**执行结果**：在确保数据被物理抹除后，自动执行解除链接操作，让文件彻底消失。

### 5. 处理只读文件
```bash
彻底删除 强制写入 系统日志.log
```
**执行结果**：即使文件权限是“只读”，它也会强行修改权限并完成粉碎任务。

## 🔍 底层逻辑详解

**彻底删除** 命令与普通删除的本质区别在于：它不是简单地移除文件名，而是直接对磁盘上的物理数据块进行覆写。

### 1. 数据覆写机制
- **普通删除 (rm)**：仅从文件系统的目录结构中摘除文件名，减少 inode 的链接计数。只要链接数不为零，数据就完好无损。
- **彻底删除 (shred)**：直接定位到文件占用的物理扇区，用随机数据或特定模式（如全0）反复覆盖原始内容。

### 2. 与硬链接的联动效应
这是一个非常关键的底层特性：**彻底删除会影响所有指向该数据的硬链接。**

- **共享本质**：硬链接之间没有“源”与“副本”之分，它们共享同一个 inode 和同一组物理数据块。
- **同步破坏**：当你对其中一个硬链接执行 `彻底删除` 时，底层的物理数据被随机乱码覆盖。此时，无论你通过哪个硬链接名字去访问，看到的都将是已被破坏的数据。
- **实验结论**：如果你有一个机密文件的多个硬链接，必须对**每一个**硬链接都执行 `彻底删除`，或者确保在执行前没有其他硬链接存在，否则数据仍可能通过其他入口被恢复（如果其他入口未被覆写）。

### 3. 为什么需要多次覆盖？
- **磁性残留**：在早期的机械硬盘上，单次覆盖可能因磁性残留而被实验室级别的设备恢复。
- **现代标准**：虽然现代硬盘密度极高，单次覆写已极难恢复，但 `彻底删除` 默认采用 3 次覆盖（符合 DoD 5220.22-M 等安全标准），以提供最高级别的安全保障。

### 4. 局限性：SSD 与现代文件系统
- **磨损均衡 (Wear Leveling)**：在固态硬盘 (SSD) 上，控制器可能会把数据写到新的物理块，导致旧块里的数据依然存在。
- **日志文件系统**：像 ext4 或 btrfs 这样的文件系统可能会在日志里留下数据的副本。
- **建议**：对于 SSD 用户，最安全的做法是使用全盘加密，然后在不需要时丢弃密钥，而不是依赖 `彻底删除`。

## 📜 Linux 中的定义与发展史

### 1. 早期 Unix 的安全需求
在 Unix 发展的早期，随着多用户系统的普及，如何确保一个用户删除的文件不会被另一个用户恢复，成为了一个严峻的安全课题。最早的 `shred` 概念源于军方对数据销毁的严格要求。

### 2. GNU Coreutils 的标准化
GNU 项目组将 `shred` 纳入 Coreutils，并为其制定了严格的覆盖算法。它不仅支持随机数覆盖，还支持符合美国国防部（DoD）标准的多种覆写模式。

### 3. 现代存储介质的挑战
随着 SSD 和云存储的兴起，`彻底删除` 的物理层假设受到了挑战。在现代 Linux 系统中，它更多被视为一种针对传统机械硬盘或加密卷内文件的有效手段。对于云环境，真正的“彻底删除”往往意味着销毁整个加密容器的密钥。
