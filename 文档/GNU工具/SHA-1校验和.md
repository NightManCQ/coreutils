# SHA-1校验和 (sha1sum)

> **对联系统理念**: 以日常汉语为导向，精准表意，规范统一。

## 📋 核心定义

**SHA-1校验和** 在汉语编程中对应英文命令：**sha1sum**。

计算并验证文件的 SHA-1（Secure Hash Algorithm 1）消息摘要。曾经广泛使用的哈希算法，现在由于安全漏洞已被SHA-256等更安全的算法取代，但仍用于遗留系统和非安全敏感场景。

**通俗理解**：
- **什么是SHA-1校验和**：就像给文件生成一个"过时的数字指纹"，`SHA-1校验和` 命令会生成一个40位的十六进制字符串（160位）。SHA-1曾经是行业标准，但在2005年发现了理论攻击，2017年实现了实际碰撞攻击。现在它主要用于遗留系统兼容性，不推荐用于新的安全应用。
- **为什么需要它**：虽然SHA-1不再安全，但仍有大量遗留系统、旧软件、Git版本控制等在使用它。了解SHA-1有助于维护旧系统、理解技术发展历史，以及在迁移到更安全算法时进行过渡。
- **应用场景**：Git版本控制、遗留系统维护、旧软件包验证、技术教育、算法迁移过渡期。

### SHA-1 特性

- **固定长度**：始终生成40位十六进制字符串（160位）
- **已不安全**：2017年实现实际碰撞攻击
- **遗留支持**：Git等工具仍在使用
- **快速计算**：比SHA-256快
- **过渡期**：正在被SHA-256取代

## 🔍 底层逻辑详解

**SHA-1校验和** 的核心逻辑是 **"实现SHA-1哈希算法"**。

### 1. SHA-1 算法概述

SHA-1由NSA设计，1995年发布，是SHA-0的改进版本。

#### 主要特点
- **输出长度**：160位（20字节）
- **块大小**：512位（64字节）
- **轮数**：80轮
- **字大小**：32位

### 2. 安全漏洞历史

#### 理论攻击
- **2005年**：王小云教授提出理论攻击
- **复杂度**：2^69次操作（低于预期的2^80）

#### 实际攻击
- **2017年**：Google SHAttered攻击
- **成果**：生成两个不同的PDF文件，SHA-1相同
- **成本**：约110年GPU计算时间

#### 现状
- **碰撞攻击**：可行
- **第二原像**：仍然困难
- **原像攻击**：仍然困难

### 3. 与 SHA-256 的对比

| 特性 | SHA-1 | SHA-256 |
|------|-------|---------|
| **输出长度** | 160位 | 256位 |
| **安全性** | 已破解 | 安全 |
| **速度** | 快 | 较慢 |
| **推荐度** | ❌ 不推荐 | ✅ 推荐 |
| **用途** | 遗留系统 | 新应用 |

### 4. Git 为何仍使用 SHA-1

#### 原因
- **非对抗性环境**：Git不是安全协议
- **偶然碰撞概率极低**：2^-160
- **迁移成本高**：需要重写整个历史
- **正在计划迁移**：Git 2.42+ 支持SHA-256

#### Git的缓解措施
- **内容验证**：结合其他机制
- **签名标签**：使用GPG签名
- **未来迁移**：计划过渡到SHA-256

### 5. 系统调用层面

```c
#include <stdio.h>
#include <openssl/sha.h>

unsigned char hash[SHA_DIGEST_LENGTH];  // 20字节
SHA_CTX ctx;

SHA1_Init(&ctx);

FILE *fp = fopen(filename, "rb");
char buffer[4096];
size_t bytes;

while ((bytes = fread(buffer, 1, sizeof(buffer), fp)) > 0) {
    SHA1_Update(&ctx, buffer, bytes);
}

SHA1_Final(hash, &ctx);
fclose(fp);

// 输出十六进制
for (int i = 0; i < SHA_DIGEST_LENGTH; i++) {
    printf("%02x", hash[i]);
}
```

#### 不应使用SHA-1的场景

## 📊 使用频率说明

在现代 Linux 系统中，`SHA-1校验和` (sha1sum) 的使用频率**中等到低**，且持续下降。

*   **原因**：
    *   SHA-1已被证明不安全。
    *   新系统普遍采用SHA-256。
    *   主要用于遗留系统维护。

*   **替代方案**：
    *   **sha256sum**：推荐的标准替代品
    *   **sha512sum**：更高安全性
    *   **b2sum**：更快的BLAKE2
    *   **编程语言库**：hashlib (Python) 等

*   **推荐使用场景**：
    *   **Git兼容性**：Git仍使用SHA-1
    *   **遗留系统**：维护旧系统
    *   **非安全校验**：内部工具、临时使用
    *   **教育目的**：讲解哈希算法演进
    *   **迁移过渡**：从SHA-1迁移到SHA-256的过程

## 📜 Linux 中的定义与发展史

### 1. SHA-1 的诞生
1995年，NSA发布了SHA-1，作为SHA-0的修复版本。

### 2. 广泛应用期
1995-2005年，SHA-1成为最流行的哈希算法，广泛用于SSL证书、数字签名等。

### 3. 安全漏洞发现
- **2005年**：理论攻击
- **2017年**：实际碰撞（SHAttered）

### 4. 逐步淘汰
- **2011年**：NIST停止接受SHA-1证书
- **2017年**：主流浏览器弃用SHA-1证书
- **2020年**：Git开始计划迁移到SHA-256

### 5. GNU Coreutils 的实现
GNU项目提供了sha1sum命令，作为coreutils包的一部分。

### 6. uutils 的 Rust 重写
uutils项目用Rust重写了sha1sum。

### 7. 现代应用生态
如今，`SHA-1校验和` 主要用于：
- **Git版本控制**：对象哈希
- **SVN**：内容寻址
- **遗留系统**：旧软件维护
- **教育**：算法演进教学

### 8. 迁移趋势
- **Git**：计划迁移到SHA-256
- **TLS**：已完全迁移到SHA-256/SHA-384
- **代码签名**：已迁移到SHA-256
- **证书**：已迁移到SHA-256

### 9. 未来的地位
SHA-1将逐渐被淘汰：
- **短期**：Git等遗留系统继续使用
- **中期**：逐步迁移到SHA-256
- **长期**：可能被SHA-3或后量子算法取代

未来5-10年内，SHA-1的使用将进一步减少，最终只在极少数遗留系统中存在。
